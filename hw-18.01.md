9. Изучите список [telegraf inputs](https://github.com/influxdata/telegraf/tree/master/plugins/inputs). 
Добавьте в конфигурацию telegraf следующий плагин - [docker](https://github.com/influxdata/telegraf/tree/master/plugins/inputs/docker):
```
[[inputs.docker]]
  endpoint = "unix:///var/run/docker.sock"
```

Дополнительно вам может потребоваться донастройка контейнера telegraf в `docker-compose.yml` дополнительного volume и 
режима privileged:
```
  telegraf:
    image: telegraf:1.4.0
    privileged: true
    volumes:
      - ./etc/telegraf.conf:/etc/telegraf/telegraf.conf:Z
      - /var/run/docker.sock:/var/run/docker.sock:Z
    links:
      - influxdb
    ports:
      - "8092:8092/udp"
      - "8094:8094"
      - "8125:8125/udp"
```

После настройке перезапустите telegraf, обновите веб интерфейс и приведите скриншотом список `measurments` в 
веб-интерфейсе базы telegraf.autogen . Там должны появиться метрики, связанные с docker.

Факультативно можете изучить какие метрики собирает telegraf после выполнения данного задания.

###  Ответ:

1. Минимальный набор метрик:
   
Инфраструктурные метрики: Загрузка CPU; Используемое дисковое пространство; Disk Write I/O (throughput & latency)

Метрики приложения: Доступность (HTTP), время ответа (p95, p99), количество ошибок (% 5xx / 4xx)

Метрики бизнес-логики: Среднее время вычислений; Длина очереди и активные задачи; Computation Success / Failure Rate

Почему:

Покрывают все критические пути: загрузка CPU → выполнение задач → запись на диск → отдача результата по HTTP. Соответствуют принципам SRE. Готов к алертингу.



2.Предложение перейти на метрики уровня сервиса (SLI/SLO).

Доступность сервиса. Если SLA 99.9%, то можно показать.

Скорость получения результата. Рост p95 — сигнал, что "сервис стал медленнее", даже если технически всё "зелёное"

Пропускная способность. Помогает планировать масштабирование.



3.Sentry (Free Tier / Self-Hosted)

Почему:

Можно установить бесплатно.

Группировка одинаковых ошибок.

Автоматическое извлечение стектрейса, переменных, пользователя, запроса

Интеграции с GitHub, Slack, Jira

Разработчики могут сами отмечать "решено", назначать ответственных



4. Вы не учитываете коды ответа 3xx (Redirection).

Правильная формула:

```

SLA = sum_2xx / sum_all

```


5. Pull-системы:

Плюсы:

Централизованное управление

Чёткая диагностика недоступности

Безопасность периметра

Service Discovery из коробки

Контроль нагрузки на источники


Минусы:

Не подходит для короткоживущих задач

Сложности с NAT/межсетевыми экранами

Ограничение масштабируемости скрейперов

Требует реализации HTTP-сервера


Push-системы: 


Плюсы:

Идеальны для эфемерных процессов

Работают через сложные сетевые тополгии

Буферизация и агрегация на стороне источника

Легче масштабировать ingest-слой

Проще интеграция с legacy



Минусы:


Проблема "Dead Man's Switch"

Сложнее централизованное управление

Риск перегрузки сервера сбора

Требует точной синхронизации времени

Сложнее верификация доступности
