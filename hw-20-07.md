## Добавим запас на отказ одной ноды (N+1)
Минимально разумная схема для обеспечения отказоустойчивости и соблюдения правила размещения реплик на разных хостах (`PodAntiAffinity`) — разделение на пулы и использование минимум 3 нод в каждом пуле.

### Stateful-пул (БД + Кеш)
Суммарные требования workload: 24 ГБ RAM, 6 CPU.  
Если взять **3 ноды**, то при падении одной ноды у нас остаётся 2 ноды.  
На эти 2 ноды должны поместиться все 24 ГБ RAM и 6 CPU + системный резерв (2 ноды × 1.5 ГБ = 3 ГБ RAM, 2 ноды × 1 CPU = 2 CPU).  
Итого требуется на 2 оставшихся нодах: 27 ГБ RAM и 8 CPU.  
Значит, каждая из 3 нод должна иметь:
- 27 / 2 ≈ 13.5 ГБ RAM (округляем до стандартного размера **16 ГБ**)
- 8 / 2 = 4 CPU (округляем до **8 CPU** для симметрии и запаса на пики)

**Итого по Stateful-пулу (3 ноды):**
- 3 × 8 CPU = 24 CPU
- 3 × 16 ГБ RAM = 48 ГБ RAM  
*Почему:* есть запас по памяти, возможна деградация при падении одной ноды с сохранением работоспособности, остаётся место для перераспределения pod’ов и соблюдения anti-affinity для 3-х реплик.

### Stateless-пул (Backend + Frontend)
Суммарные требования workload: 6.25 ГБ RAM, 11 CPU.  
CPU — основной лимитирующий фактор.  
Если взять **3 ноды**, то при падении одной ноды у нас остаётся 2 ноды.  
На эти 2 ноды должны поместиться 11 CPU и 6.25 ГБ RAM + системный резерв (2 CPU, 3 ГБ RAM).  
Итого требуется на 2 оставшихся нодах: 13 CPU и ~9.25 ГБ RAM.  
Значит, каждая из 3 нод должна иметь:
- 13 / 2 = 6.5 CPU (округляем до стандартного размера **8 CPU**)
- 9.25 / 2 ≈ 4.6 ГБ RAM (округляем до стандартного размера **16 ГБ** для унификации и запаса под HPA)

**Итого по Stateless-пулу (3 ноды):**
- 3 × 8 CPU = 24 CPU
- 3 × 16 ГБ RAM = 48 ГБ RAM

## Что обеспечивает такая схема
- **Отказоустойчивость**: переживает падение одной ноды в каждом пуле без нарушения работы критичных сервисов.
- **Распределение нагрузки**: корректное разделение stateful-нагрузки с учётом требований к дисковой подсистеме.
- **Изоляция**: БД и кеш изолированы от backend/frontend (предотвращение эффекта "шумных соседей").
- **Масштабируемость**: возможность горизонтального масштабирования (HPA) в пределах выделенных ресурсов.
- **Гибкость**: возможность управления размещением через `nodeSelector`, `taints` и `tolerations`.

---

## Архитектурные решения

### Для БД и кеша:
- `StatefulSet` (для стабильных сетевых идентификаторов и порядка запуска).
- `PersistentVolumeClaim` (PVC) для сохранения данных.
- `PodDisruptionBudget` (PDB) для контроля количества доступных подов при обслуживании нод.
- `PodAntiAffinity` (`required`) для гарантированного размещения реплик на разных физических нодах.
- Выделенный `nodeSelector` / `taints` для изоляции пула.

### Для backend и frontend:
- `Deployment` для быстрого масштабирования и обновлений.
- `HorizontalPodAutoscaler` (HPA) для реакции на рост нагрузки.
- `PodDisruptionBudget` (PDB) для обеспечения доступности во время обновлений.
- `readiness` / `liveness` probes для контроля здоровья приложения.
