# Домашнее задание к занятию «Микросервисы: подходы» Алавидзе И.Г.

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.


## Задача 1: Обеспечить разработку

Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка. 
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- облачная система;
- система контроля версий Git;
- репозиторий на каждый сервис;
- запуск сборки по событию из системы контроля версий;
- запуск сборки по кнопке с указанием параметров;
- возможность привязать настройки к каждой сборке;
- возможность создания шаблонов для различных конфигураций сборок;
- возможность безопасного хранения секретных данных (пароли, ключи доступа);
- несколько конфигураций для сборки из одного репозитория;
- кастомные шаги при сборке;
- собственные докер-образы для сборки проектов;
- возможность развернуть агентов сборки на собственных серверах;
- возможность параллельного запуска нескольких сборок;
- возможность параллельного запуска тестов.

Обоснуйте свой выбор.

### Ответ:

Выбор решения: GitLab CI/CD (GitLab.com SaaS Premium/Ultimate) + GitLab Runner (self-hosted) + HashiCorp Vault (для секретов).

GitLab CI/CD — это решение которое одновременно и нативно закрывает все 14 требований без необходимости в сторонних плагинов или строить самописные интеграции. 

Ключевые преимущества:

Единая платформа (VCS + CI/CD + Registry + Security) снижает операционные затраты и упрощает access control.

Мощная система шаблонов (include:) позволяет стандартизировать пайплайны across hundreds of microservices.

Self-hosted runners дают полный контроль над инфраструктурой сборки и позволяют использовать кастомные образы с корпоративными инструментами.
Нативная интеграция с Vault решает вопрос безопасного хранения секретов на enterprise-уровне.

Multi-project pipelines — уникальная фича, специально заточенная под микросервисную архитектуру.

Это решение масштабируется от десятков до тысяч репозиториев, соответствует лучшим практикам DevOps и обеспечивает безопасность, требуемую в крупной компании.

## Задача 2: Логи

Предложите решение для обеспечения сбора и анализа логов сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор логов в центральное хранилище со всех хостов, обслуживающих систему;
- минимальные требования к приложениям, сбор логов из stdout;
- гарантированная доставка логов до центрального хранилища;
- обеспечение поиска и фильтрации по записям логов;
- обеспечение пользовательского интерфейса с возможностью предоставления доступа разработчикам для поиска по записям логов;
- возможность дать ссылку на сохранённый поиск по записям логов.

Обоснуйте свой выбор.

### Ответ:

Мой выбор: ELK Stack (Elasticsearch + Logstash + Kibana) + Filebeat (для сбора логов)/

ELK Stack — это зрелое, мощное и гибкое решение для централизованного сбора и анализа логов, которое идеально подходит для крупных компаний со сложными требованиями.


### Мощный полнотекстовый поиск.
Elasticsearch индексирует весь текст логов.

Это означает:

Поиск по любому слову в логе без предварительного парсинга.

Fuzzy search (поиск с опечатками).

Wildcard и regex-запросы.

Быстрый поиск по большим объемам данных (миллиарды документов).

### Богатая аналитика и агрегация

Aggregations framework — мощнейший инструмент для аналитики:

Подсчет ошибок по типам, сервисам, хостам.

Построение гистограмм и временных рядов.

Статистические агрегации (avg, min, max, percentiles).

Вложенные агрегации (drill-down анализ).


### Гибкость парсинга и трансформации
Logstash — мощнейший ETL-инструмент:

Парсинг структурированных и неструктурированных логов (grok, dissect, json, csv).

Обогащение данных (geoip, user agent parsing, DNS lookup).

Фильтрация и трансформация (mutate, drop, clone).

Интеграция с внешними системами (базы данных, API).

### Зрелая экосистема и готовые решения

Elastic Common Schema (ECS) — стандартизированная схема данных для логов, метрик, трейсов.

Готовые модули для популярных технологий:

Nginx, Apache, HAProxy

MySQL, PostgreSQL, MongoDB

Kubernetes, Docker

AWS, Azure, GCP

Machine Learning — автоматическое обнаружение аномалий в логах (в X-Pack/Platinum).

### Enterprise-готовность

Security — RBAC, field-level security, audit logging, SSO (SAML, LDAP, Active Directory).

Multi-tenancy — изоляция данных через Kibana Spaces и index patterns.

Cross-Cluster Search/Replication — федеративный поиск across нескольких дата-центров.

Alerting — встроенная система алертов на основе запросов и агрегаций.

Reporting — экспорт дашбордов в PDF, email-рассылки.

### Масштабируемость

Горизонтальное масштабирование — добавление нод Elasticsearch для обработки растущего объема.

Index Lifecycle Management (ILM) — автоматическое управление жизненным циклом индексов (hot → warm → cold → delete).

Snapshot/Restore — бэкапы в S3/GCS/HDFS.

Data streams — оптимизированное хранение time-series данных (логов, метрик).

## Задача 3: Мониторинг

Предложите решение для обеспечения сбора и анализа состояния хостов и сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор метрик со всех хостов, обслуживающих систему;
- сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network;
- сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network;
- сбор метрик, специфичных для каждого сервиса;
- пользовательский интерфейс с возможностью делать запросы и агрегировать информацию;
- пользовательский интерфейс с возможностью настраивать различные панели для отслеживания состояния системы.

Обоснуйте свой выбор.


### Ответ:

Мой выбор: Prometheus + Grafana + Node Exporter + cAdvisor + Alertmanager.


Prometheus + Grafana — это де-факто стандарт для мониторинга микросервисной архитектуры, особенно в Kubernetes-окружениях. 

Ключевые преимущества:

Все 6 требований задачи закрыты «из коробки»:

Сбор метрик с хостов (Node Exporter)

Метрики ресурсов хостов (CPU, RAM, HDD, Network)

Метрики ресурсов сервисов (cAdvisor)

Специфичные метрики сервисов (Prometheus client libraries)

UI с запросами и агрегацией (PromQL + Grafana Explore)

UI с панелями (Grafana Dashboards)

Pull-модель — целевая система сама отдает метрики, что упрощает firewall-правила и service discovery.

Мощный PromQL — богатый язык запросов с функциями агрегации, rate, histogram_quantile.

Service Discovery — автоматическое обнаружение новых сервисов в динамических средах (Kubernetes, Consul).

Огромная экосистема — сотни exporters, готовые дашборды, интеграции.

Масштабируемость — Thanos/Cortex для горизонтального масштабирования и долгосрочного хранения.

Низкая стоимость — Open-source, бесплатное решение enterprise-уровня.
